## 二、P1 级：建议修复

### 2.1 [AQR] 缺少因子正交化流程

**当前问题**：当同时评测高度相关的因子（如不同窗口的动量 STR_5d vs STR_20d），当前系统各自独立评测，无法区分各自的独立预测贡献。

**影响**：多个进入 core 等级的因子可能本质上是在捕捉同一个信号，造成多因子组合中的冗余。

**建议方案**：

`
选项 A（推荐）：在 evaluate() 中增加条件正交化步骤
    1. 计算因子与已有 core 级别因子的截面相关系数
    2. 对高相关因子（|r|>0.7）做 Gram-Schmidt 正交化
    3. 输出正交化前/后的 IC 对比

选项 B（简化）：在评测报告中增加因子相关性分析图表
    从 registry.json 读取已有 core 因子
    计算当前因子与已入库因子的时间序列 IC 相关性
`

### 2.2 [Renaissance] 缺少未来函数显式检测

**当前问题**：没有自动化机制来检测因子计算中是否存在前向偏差。常见的未来函数包括：

- 使用了未来才知道的数据（如财报发布后的修正数据而非发布前的数据）
- 复权因子使用了未来拆股/分红信息（rgMax(adj_factor) 策略可能引入）
- 标签计算中使用了未来退市股票数据

**影响**：某些因子可能看起来表现优异，但实际上是因为计算中混入了未来信息。

**建议方案**：

`python

    '''检测因子值中可疑的未来信息'''
    # 1. 检查因子值与次日收益率的异常高相关性（>0.99 说明可能用了未来close）
    # 2. 检查停牌日是否有非空因子值（停牌日不应有交易数据）
    # 3. 检查复权因子跳跃日（分红/送股日）因子值是否异常
`

### 2.4 [Two Sigma] 尾部风险指标缺失

**当前问题**：分层回测只报告了多空组合的均值收益和夏普比率，没有报告：

- 第 1 分位（最差组合）的 CVaR 95%
- 多空组合的最大回撤（Max Drawdown）和回撤恢复期
- 多空组合的 Skewness / Kurtosis

**影响**：一个高正偏度的因子可能年化夏普通过，但尾部风险极大（如末日虚值期权策略）。

**建议方案**：

`python
# 在 calc_ls_stats() 中增加尾部指标
def calc_ls_stats(ls_df, ...):
    returns = ls_df['long_short']
    return {
        ...
        'max_drawdown': calc_max_drawdown(returns),
        'cvar_95': returns[returns <= returns.quantile(0.05)].mean(),
        'skewness': returns.skew(),
        'kurtosis': returns.kurtosis(),
    }
`

## 三、P2 级：值得改进

### 3.1 [Architect] 批量评测缺少编排策略

当前 evaluate_batch() 只是简单的 for 循环逐个评测。缺少：
- 评测队列优先级排序（core candidate > satellite > archive）
- 批量评测的并发控制（大因子库全量评测耗时可能 >1 小时）
- 评测中间结果的持久化（中途中断需从头再来）

### 3.2 [Millennium] 缺少评测结果缓存

当前每次 evaluate() 都会全量重算，包括时间固定的 IC/分层收益指标。同一个因子同一个时间段多次评测产生的结果是确定性的，应该缓存。

**建议方案**：在 evaluate() 返回结果前，计算结果的哈希值存入 egistry.json。下次相同因子 + 相同时间段 + 相同配置时直接返回缓存。

### 3.3 [Point72] 缺少 Purged K-Fold CV

Walk-Forward 已经解决了时序过拟合问题，但缺少更细致的 **Purged Cross-Validation**——训练集和验证集之间没有足够"净化期"（purging），紧邻的数据点可能因时间自相关而造成信息泄漏。

**建议**：在 walk_forward_evaluate() 的方法增加 purge_days=20 的参数，在每个训练/验证窗口之间插入 20 个交易日的净化期。

## 四、NOTE 级：远期规划

### 4.1 [Citadel] 信号半衰期自动估算

当前衰减分析使用固定的 [1, 5, 10, 20, 40, 60] horizon 集。不同因子的信号持久性差异很大——换手率因子的半衰期可能只有 2-3 天，而质量因子可能持续数月。

**方案**：增加自动半衰期估算函数，用指数衰减模型拟合 IC(h) = IC₀·exp(-h/τ)，输出 τ 值。

### 4.2 [D.E.Shaw] 配对交易/协整评价缺失

对于均值回归类因子（反转、波动率等），缺少：
- 配对交易的 ADF 协整检验
- Ornstein-Uhlenbeck 过程的半衰期估计（θ 参数）
- Hurst 指数（判断序列是均值回归还是趋势）

**方案**：新增 statarb_analysis.py 模块，对反转类因子专项检验。

### 4.3 [Bridgewater] 市场状态划分

因子在不同市场状态下的表现可能截然相反（如动量在牛市强、在震荡市弱）。缺少按市场状态分层的评测。

**方案**：

`python
# 在 robustness.py 中增加 regime-aware 分析
REGIME_DEF = {
    'bull': 'MKT > 0 且波动率低',
    'bear': 'MKT < -0.02',
    'choppy': 'MKT 接近 0 且高换手',
}
`

### 4.4 [Virtu] 执行模型精细化

当前入场/出场价使用 open/close，滑点为线性 10bp 假设。理想方案：
- 使用 VWAP（成交量加权均价）替代 open/close
- 冲击成本模型区分大盘股（低冲击）vs 小盘股（高冲击）
- 增加最小交易量约束（单笔交易不低于 1 手）

### 4.5 [Man Group] 多因子合成接口

当前单因子评测系统的输出（IC、分层收益等）没有标准化的接口供下游多因子合成模块消费。

**方案**：定义 FactorEvaluationResult Protocol 或 dataclass，包含评测时间戳、IC/ICIR、最佳持有期、资金容量等级等字段，作为多因子组合的标准输入。

### 4.6 [Point72] 特征重要性分析

单因子评测与 ML 模型的 SHAP 值分析没有打通。如果后续要训练 LightGBM/XGBoost 模型，评测系统的输出应该能直接作为特征筛选的输入。

**方案**：在评测报告中输出因子在常用 ML 模型下的单变量 AUC / Feature Importance 热力图。

### 4.7 [Jane Street] 微观结构因子

L2 级行情数据（买卖盘口深度、价差、订单流不平衡）在当前系统中没有任何接入。对高频因子（如盘口斜率、订单簿压力）的评测完全缺失。

**方案**：需要额外接入 L2 行情数据源（如聚宽 Tick 级数据），属于较大的架构扩展。

### 4.8 [Compliance] 阈值自动校准

当前等级判定的所有阈值（IC>0.02, ICIR>0.4 等）都是经验值。随着数据周期扩展到 2010-2026，这些阈值可能需要按市场环境做分位数校准。

**方案**：

`python
def calibrate_thresholds(all_factors_results: list, percentile: float = 0.3):
    '''从全因子库的评测结果自动校准 IC/ICIR 阈值'''
    ic_distribution = [r['ic_stats']['ic_mean'] for r in all_factors_results]
    return np.percentile(ic_distribution, percentile)
`
